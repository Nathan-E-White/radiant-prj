# Retain Repository Verification As The Durable Capability-Contract Index

| Field | Value |
| --- | --- |
| Document ID | ADR-0008 |
| Revision | 1.0 |
| Status | Accepted |
| Owner | Software |
| Baseline | Capability lifecycle documentation |

## Status

Accepted

## Context

The repository already contains executable repository-verification claims and active browser- and Docker-delivery verification modules. Their contracts are distributed across scripts, manifests, controlled procedures, and CI configuration. A maintainer can verify a claim, but previously had to reconstruct why a module exists, which evidence is authoritative, and whether an older experiment remains active from commit history.

The quality plan, configuration-management procedure, requirements-to-verification matrix, verification plan, and corrective-action procedure are the controlled system records. Capability documentation must point to those records rather than reproduce or supersede them.

## Decision

Treat repository verification as the durable index of repository-owned capability contracts. Each retained verification or delivery module shall have concise module documentation that names its purpose, owned contract, highest verification seam, evidence source, owner role, and lifecycle state. The documentation shall link to the controlled system records and to the module's decision record.

The executable inventory in `config/repository-verification.json` is the retained presence check for these records. It proves that the decision records and module documentation exist; it does not replace their review or prove that the commands pass.

## Lifecycle Direction

Active modules are maintained implementation and verification capability. Retired experiments remain historical context only: they retain their Git history and may explain a decision, but must not be cited as current architecture, delivery, or verification evidence. Work under reconciliation is explicitly marked as such and cannot be presented as active proof until its contract and evidence source are reconciled.

## Consequences

The repository gains a stable route from a verification claim to its decision, module contract, current evidence, and lifecycle status. This adds a small documentation-maintenance obligation when a module is introduced, retired, or materially re-scoped. The alternative is repeated archaeological work, a field with surprisingly poor unit economics.
