# Govern Backend-Backed Fleet Board Execution And Read Boundaries

| Field | Value |
| --- | --- |
| Document ID | ADR-0007 |
| Revision | 1.0 |
| Status | Accepted |
| Owner | Software |
| Baseline | Fleet Board V3 backend-backed horizon |

## Status

Accepted

## Context

Fleet Board V2 deliberately keeps Simulation Jobs, Simulation Budget, Simulation Container Tokens, and Insight Tokens in the local deterministic game reducer. ADR-0006 deferred backend game events, persistence, live ingestion, and Artifact Forge until a separate decision could preserve the stream separation accepted in ADR-0005.

Fleet Board V3 may now request narrowly bounded backend work and consume coherent read models. This authority must not turn a local game action into infrastructure control, a simulation result into a measurement, or a browser into a data-plane client.

## Decision

Accept a mediated backend-backed horizon under the boundaries below. The local Fleet Board reducer remains authoritative for local game rules. A backend game-intent API may accept only named, validated intents; Simulation Ops, projection, twin, and read-model modules remain authoritative for their own lifecycles.

ADR-0005 remains in force. Measured State, operational telemetry, Simulated Result State, Twin State, and Lineage retain separate contracts, storage paths, and Value Basis semantics.

## Backend Execution Boundary

The only Fleet Board intent authorized to request a SimOps Run is an explicit `requestArtifactForge` intent. It must reference one game session, one stable reactor identity, one completed local Simulation Job, one supported simulation recipe, and one client-generated idempotency key. The player must request it deliberately; a day tick or Simulation Job completion must not launch backend work automatically.

The backend validates the intent, records it, and may create or associate exactly one SimOps Run. The Simulation Job remains local game state. It is not promoted into, replaced by, or lifecycle-coupled to the SimOps Run. Rejection, Run failure, and recovery leave the local job intact and produce an explicit game-facing request outcome.

An artifact may become a game outcome only when all of these conditions hold:

- its SimOps Run reached the accepted successful terminal state;
- its artifact kind, schema version, and simulation recipe are on the server-side eligibility allowlist;
- it contains public-safe Simulated Result State rather than operational telemetry;
- its Lineage identifies the game session, reactor, SimOps Run, model or recipe, and artifact;
- its integrity and completeness checks pass; and
- the versioned outcome mapping has not already consumed it.

Operational telemetry, incomplete artifacts, missing Lineage, failed or cancelled Runs, Twin State by itself, and artifacts with a measured or imputed Value Basis are ineligible. An eligible artifact may produce at most one named, versioned game outcome. Artifact Forge records the eligibility decision and mapping; it does not declare the artifact objective evidence or a safety result.

## Stream And Identity Boundary

The following identities and lifecycles remain distinct:

- a Simulation Job is a deterministic local-game lifecycle;
- a SimOps Run is a backend execution and artifact lifecycle;
- a Resident Source is a long-running public-safe measured-source stand-in;
- a Run-Scoped Simulation Worker belongs to one SimOps Run and produces operational telemetry or Simulated Result State;
- operational telemetry describes worker or Run operation and is never a game reward source;
- Simulated Result State is run-scoped model output with `valueBasis=simulated`;
- Twin State is the twin projector's current model-derived state, including Imputed State;
- Lineage explains the source, run, model, artifact, and projection relationships.

Backend records use opaque `gameSessionId`, `gameIntentId`, and `reactorId` values. SimOps retains its own `runId`, worker identity, and artifact identity. Resident Sources retain source identity. Joins are explicit association records; no identifier is reused across domains.

A Reactor Telemetry Worker Set is one bounded group of Resident Sources for one player-added reactor. It is not a Run-Scoped Simulation Worker set. One game session may own at most four dynamic reactors, each reactor may own one active worker set, and a set may contain at most three Resident Source workers. One reactor may have at most one active Artifact Forge request, with at most four active requests per game session. Server configuration may lower these caps but may not raise them without revising this decision.

## Persistence And Lifecycle Boundary

The backend persists accepted and rejected game intents, idempotency keys, domain associations, eligibility decisions, and applied outcomes in a game-event ledger. It does not persist the reducer's full local Simulation Job state as a SimOps record. Applying an Artifact Forge outcome is idempotent and atomic with recording its consumption marker.

Inactive game sessions expire after 24 hours. Intent, association, eligibility, and outcome records are retained for seven days after session expiry or until a Configured Data Flush, whichever comes first. SimOps Runs, workers, and artifacts retain their existing Run lifecycle, cleanup policy, and retention; a game association never extends them. A Reactor Telemetry Worker Set exists only while its reactor and session are active. Removal or expiry revokes its ingest credentials immediately and starts bounded cleanup, which must finish or expose a retryable failed-cleanup state within five minutes.

Retries reuse the original idempotency key and domain identities. A retry must recover the recorded request, Run association, eligibility decision, or applied outcome rather than create a parallel lifecycle. Failed launch, persistence, publication, cleanup, and eligibility paths remain visible through stable terminal or retryable states.

## Security Boundary

Ordinary Run-Scoped Simulation Workers retain Gateway-Only Worker Ingest. They receive a run-scoped gateway URL and token and no Redpanda, Postgres, Iceberg, object-store, Docker, Kubernetes, Slurm, or cluster credentials.

Each Resident Source worker in a Reactor Telemetry Worker Set receives only a source-scoped, reactor-bound gateway ingest credential. It receives no broker, database, lake, container, runtime, or cluster credential. Artifact Forge and game-intent services use server-side interfaces; credentials do not pass through game events or Lineage.

Browser code may call an authenticated game-intent interface and a read-only Workbench interface. It receives no ingest token and no broker, database, lake, container, runtime, or cluster credential. Fleet Board actions may select a supported recipe but may not choose runtime adapters, images, worker counts, credentials, retention, or cleanup policy.

## Configured Data Flush And Recovery

Configured Data Flush is a dry-run-first, targeted local-demo operation. Its plan identifies the current data generation, every runtime record class to clear, the next generation, and every protected resource before mutation.

The flush may clear game-event ledger records, telemetry and result records, Twin State and Lineage records, projection read models, and expired Run or artifact records covered by their retention policy. It preserves schemas, Resident Source declarations, credentials and credential configuration, required topics, Compose wiring, platform configuration, and protected volumes. It must not use environment teardown, generic volume pruning, volume-wide deletion, or `docker compose down -v`.

The flush creates a new monotonic data generation before accepting subsequent writes. Recovery either completes that generation transition or resumes the prior generation; it never publishes a mixture as one Workbench Snapshot. A subsequent Run must start without platform reprovisioning.

## Live Read Boundary

The read-only Workbench interface returns one coherent Workbench Snapshot containing a generation marker and independently labeled Measured State, Simulated Result State, Twin State, and Lineage. A refresh replaces the prior Snapshot only after the new generation is complete. Partial failure leaves the last coherent Snapshot visible with explicit stale or recovery state.

Fixture fallback is explicit and whole-Snapshot. The frontend may select a fixture Snapshot only under the accepted fallback policy; it must not fill missing fields in a live Snapshot with fixture data or combine generations. Recovery replaces the fixture Snapshot with one coherent live Snapshot.

## Consequences

Fleet Board may gain backend-backed outcomes without becoming a scheduler or telemetry console. The cost is explicit identity, persistence, recovery, retention, and sequencing policy across several modules. Those constraints are intentional: the backend can enrich the game, but it cannot quietly redefine the provenance of engineering data or hand infrastructure authority to the browser.
