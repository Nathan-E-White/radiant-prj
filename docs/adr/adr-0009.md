# Keep Browser And Docker Delivery Verification As Separate Active Capabilities

| Field | Value |
| --- | --- |
| Document ID | ADR-0009 |
| Revision | 1.0 |
| Status | Accepted |
| Owner | Software |
| Baseline | Capability lifecycle documentation |

## Status

Accepted

## Context

Browser delivery and Docker packaging/delivery have different release risks and evidence seams. Browser delivery establishes a production frontend bundle and output budgets. Docker packaging establishes build-context, image-content, image-size, and delivery-target budgets. Treating either as a generic build check obscures the contract that a failed claim is intended to protect.

The former Randiant Runtime sandbox was a separate infrastructure experiment and was retired in Issue #140. Its history may inform retrospective analysis, but it is not an active delivery or verification module.

## Decision

Retain Browser Delivery Verification and Docker Packaging/Delivery Verification as separate active modules under the Repository Verification capability. Each owns its own documented contract and highest verification seam while participating in the common repository-verification inventory and controlled quality records.

No active module may borrow the retired sandbox as current evidence. Any future replacement or incomplete migration shall be marked under reconciliation until its own decision, module record, evidence source, and lifecycle state are present.

## Consequences

Maintainers can distinguish browser-output regressions from container-delivery regressions and locate the evidence appropriate to each. The cost is intentionally explicit module documentation; a single cheerful green build would be cheaper, but less informative when it matters.
