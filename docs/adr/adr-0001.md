## Status
Accepted

## Teaser
This panel simulates how an engineer orchestrates a tightly coupled, multi-app nuclear simulation. For example, coupling an open-source tool like the OpenMC Monte Carlo Code for neutronics with Idaho National Lab's MOOSE Framework for thermal-hydraulics.

* **🧪 Nuclear Context** Simulating full-core 3D thermal gradients and neutron fluxes simultaneously.
* **🖥️ HPC Infrastructure Metrics** Slurm job allocation statuses, MPI rank distributions per application (e.g., 512 ranks for neutronics, 256 for fluids), and internode synchronization latency.
* **📊 Visual Elements** Live timeline displaying MPI barrier wait times alongside visualizations such as Gantt charts of active Slurm queues.

## Context
The SimEngine Operator Console requires real-time layout rendering of multiphysics cluster data (Slurm jobs, 512+ MPI ranks from OpenMC, and 256+ ranks from MOOSE Framework) alongside streaming telemetry. Traditional web applications face a critical bottleneck: running high-frequency data ingestion (20Hz) on the browser main thread triggers heavy Document Object Model (DOM) layouts, causing micro-stutters, lagging inputs, and high CPU thread usage. In safety-critical Hardware-in-the-Loop (HIL) testing environments, UI rendering delays are unacceptable.

## Decision
We will decouple data ingestion from the visualization layer by routing incoming telemetry frames through an asynchronous, persistent TCP WebSocket pipeline directly into a dedicated WebGL canvas wrapper (via Phaser/PixiJS engine layout segments).

Numerical string updates (such as changing Slurm rank integers) are written instantly to isolated DOM text blocks via specific structural IDs, while the heavy processing loops (such as core thermal gradients and fluid flows) are offloaded onto local GPU graphics pipelines at a fixed 60 frames per second.

```text
[Go Backend Stream] ---> (WebSocket) ---> [Browser Event Worker Thread]
                                               |
                     +-------------------------+-------------------------+

                     | (Text Nodes Only)                                 | (WebGL Data Buffers)
                     v                                                   v
         [Lean Static Tailwind DOM]                             [Phaser GPU Context Canvas]
         (Slurm Queue State Strings)                            (Fluid Flow Paths / Core Mesh)
```

## Consequences
### Positive

#### Main Thread Isolation
Eliminates layout re-flow logic. Browser compute stays low even under severe failure transients.

#### Frame Stability
Visual animation streams lock smoothly at 60 FPS without dropping frames or blocking user control sliders.

#### React Bypass Option
Can bypass React hooks where a dedicated rendering loop is appropriate.

### Negative

#### Architectural Complexity
Requires maintaining a dedicated server-side WebSocket data streaming broker (Go binary stack) rather than consuming a simple REST API.

#### Text Field Limitations
Phaser canvas engines render numerical tables awkwardly, requiring careful CSS overlay framing.
