# Restrict Gateway Execution And Credential Authority

| Field | Value |
| --- | --- |
| Document ID | ADR-0012 |
| Revision | 1.0 |
| Status | Accepted |
| Owner | Software |
| Baseline | Gateway execution boundary |

## Status

Accepted

## Context

The gateway accepts bounded Run and job requests, validates client identity, selects an execution adapter, and receives worker ingest. It is therefore the point where a browser-facing request could otherwise become scheduler, broker, database, container, or cluster authority.

ADR-0007 applies this concern to Fleet Board intents. The gateway boundary applies more broadly to every direct client and worker interaction, including the mock and opt-in `sbatch` paths.

## Decision

The gateway is the sole authority that translates an authenticated, validated request into execution work. Client-facing execution requests require the configured mTLS identity and accepted request bounds. Mock execution is the local default; `sbatch` execution is opt-in and may invoke only configured, allowlisted work scripts with bounded arguments and timeouts.

Browsers receive read-only results, explicit request outcomes, and short-lived subscription metadata where applicable. They receive no scheduler, Docker, Kubernetes, Redpanda, Postgres, Iceberg, object-store, or worker-ingest credential. A worker receives only its scoped gateway-ingest credential and the role-specific configuration established under ADR-0010; it does not receive the data-plane or runtime-management credentials.

Gateway failures remain explicit. Authentication, authorization, validation, adapter, and worker-ingest failures must not fall back to a more privileged path, inferred fixture, or unbounded command execution.

## Consequences

The gateway remains a narrow execution and credential seam rather than a pass-through to infrastructure. That concentrates audit, validation, and failure handling in one module, at the ordinary price of requiring adapter and client work to stay within the published contract. The current external surface is recorded in the [Interface Control Document](../design/interface-control.md).
